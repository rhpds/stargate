## Admin API Key

Many admin endpoints require authentication via the `X-API-Key` header. The key
is set during deployment (generated by `deploy-infra01.sh` or stored in the
`stargate-secrets` OpenShift secret).

Usage:

```bash
curl -H "X-API-Key: <your-admin-key>" \
  https://stargate-api.apps.cluster.example.com/admin/scheduler/start
```

Requests from allowed CORS origins (the dashboard) bypass API key checks for
read-only browser requests.

## Deployment

### Prerequisites

- `oc` CLI logged into ocpv-infra01
- Kubeconfig files in `secrets/` with valid service account tokens
- LiteLLM API key (for LLM features)
- `podman` for image builds

### Full Deployment to infra01

```bash
# Dry run first to verify
./scripts/deploy-infra01.sh --dry-run

# Build images and deploy
./scripts/deploy-infra01.sh --build
```

The script performs these steps:

1. Verifies cluster connection to infra01
2. Creates the `stargate` namespace
3. Creates the `stargate-kubeconfigs` secret from `secrets/` directory
4. Creates the `stargate-secrets` secret (PostgreSQL password, LiteLLM key, admin API key)
5. Sets up OAuth
6. Builds and pushes three container images to the internal registry (if `--build`)
7. Deploys via Helm chart (`deploy/helm/stargate/`)

### Build Images Only

```bash
# API server
make podman-build

# Scanner (includes oc binary)
make podman-build-scanner

# Frontend
make podman-build-frontend

# All scanner + frontend
make podman-build-all
```

### Local Development

```bash
# Start PostgreSQL
podman-compose -f podman-compose.yml up -d postgres

# Run API server with hot reload
make api-local

# Run frontend dev server (separate terminal)
cd frontend && npm run dev
```

The API listens on port 8090. The frontend dev server proxies API requests to
`http://localhost:8090`.

## Start/Stop Scanners

### Via API (Recommended)

```bash
# Start all scanner workers
curl -X POST -H "X-API-Key: <key>" \
  "http://localhost:8090/admin/scheduler/start?tier1=300&tier2=900&tier3=3600&batch=5"

# Stop all scanner workers
curl -X POST -H "X-API-Key: <key>" \
  http://localhost:8090/admin/scheduler/stop

# Check scanner status
curl http://localhost:8090/admin/scheduler/status
```

### Via CLI (Standalone)

```bash
# All clusters with default intervals
python3 -m cli.scheduler --api-url http://localhost:8090

# Specific clusters
python3 -m cli.scheduler --api-url http://localhost:8090 \
  --clusters ocpv06,ocpv07,ocpv08

# Custom intervals
python3 -m cli.scheduler --api-url http://localhost:8090 \
  --tier1 300 --tier2 900 --tier3 3600 --batch 5

# Single cluster worker
python3 -m cli.worker --cluster ocpv06 --api-url http://localhost:8090

# One-shot scan (no persistence)
python3 -m cli.scan --cluster ocpv06
```

### Via podman-compose

```bash
podman-compose -f podman-compose.yml up -d scheduler babylon-worker
podman-compose -f podman-compose.yml logs -f scheduler
podman-compose -f podman-compose.yml logs -f babylon-worker
```

### Tier Intervals

| Parameter | Default | Description                              |
|-----------|---------|------------------------------------------|
| `tier1`   | 300s    | Node metrics (oc adm top nodes)          |
| `tier2`   | 900s    | Pod delta scan (oc get pods -A)          |
| `tier3`   | 3600s   | Namespace evidence (batched, rotated)    |
| `batch`   | 5       | Namespaces per Tier 3 cycle              |

Workers stagger by 30 seconds between clusters. The Babylon control plane
worker starts after all cluster workers and runs every 3 minutes.

## Switch Evidence Modes

StarGate can use real cluster evidence or synthetic evidence from the emulator.

### Set to Synthetic

```bash
curl -X POST http://localhost:8090/admin/evidence-source \
  -H "Content-Type: application/json" \
  -d '{"source": "synthetic", "scenario": "gaudi_saturation"}'
```

Available scenarios: `healthy_baseline`, `gaudi_saturation`, `node_failure`,
`provision_blocked`, `memory_pressure`, `mixed_contention`, `xeon_underutil`.

### Set Back to Real

```bash
curl -X POST http://localhost:8090/admin/evidence-source \
  -H "Content-Type: application/json" \
  -d '{"source": "real"}'
```

### Check Current Source

```bash
curl http://localhost:8090/admin/evidence-source
```

## Enable/Disable Dry-Run

Dry-run mode prevents the action executor from performing any write operations.
When enabled, actions are logged and audit entries are written with status
`skipped_dry_run`.

### Enable Dry-Run

```bash
curl -X POST http://localhost:8090/admin/dry-run \
  -H "Content-Type: application/json" \
  -d '{"enabled": true}'
```

### Disable Dry-Run

```bash
curl -X POST http://localhost:8090/admin/dry-run \
  -H "Content-Type: application/json" \
  -d '{"enabled": false}'
```

Dry-run can also be set at startup via the `STARGATE_DRY_RUN=true` environment
variable.

## Run Validation

The validation endpoint runs all 7 synthetic scenarios and compares outcomes
against expected results.

```bash
curl -X POST http://localhost:8090/admin/validate
```

Response:

```json
{
  "total": 7,
  "passed": 7,
  "failed": 0,
  "results": [
    {
      "scenario": "gaudi_saturation",
      "match": true,
      "expected_recommendations": ["cluster_capacity"],
      "stages_evaluated": 9,
      "mismatches": []
    }
  ]
}
```

The emulator project (`stargate-synthetic-client-emulator/`) must be present
as a sibling directory. If not found, the endpoint returns an error.

## Generate Receipts

Phase gate receipts are structured JSON files that provide machine-readable
evidence of each gate's pass/fail status.

```bash
python3 scripts/generate-receipts.py
```

This runs:

1. **Test summary** -- All test suites across platform, emulator, and mock cluster
2. **Phase A** -- Shadow mode feedback loop (7 scenarios resolve after action)
3. **Phase B** -- Mock cluster command validation
4. **Phase C** -- Real execution on `stargate-test` namespace
5. **Rubric matrix** -- 7 scenarios x 11 stages = 77 cells
6. **Substrate routing** -- 7 scenarios x Gaudi/Xeon6 decisions
7. **Infrastructure** -- Deployment inventory

Receipts are saved to `receipts/` directory. An `overall-summary.json` is
generated that aggregates all gate results.

## Database Backup/Restore

### Backup

```bash
# Manual backup
./scripts/db-backup.sh

# Or via make
make db-backup
```

Backups are saved to `backups/stargate-YYYYMMDD-HHMMSS.sql.gz`. The script
uses `pg_dump` from the Podman PostgreSQL container and automatically prunes
backups older than 7 days.

### Restore

```bash
# List available backups
./scripts/db-restore.sh

# Restore a specific backup
./scripts/db-restore.sh backups/stargate-20260511-120000.sql.gz

# Or via make
make db-restore FILE=backups/stargate-20260511-120000.sql.gz
```

The restore script prompts for confirmation before overwriting data. It uses
`gunzip` piped to `psql` in the Podman container.

### Database Migration

Migrations are managed by Alembic. On startup, `init_db()` runs
`alembic upgrade head` automatically. If Alembic fails, it falls back to
`Base.metadata.create_all()`.

```bash
# Manual migration
alembic upgrade head

# Create new migration
alembic revision --autogenerate -m "description"
```

### Evaluation (Feedback Loop Quality)

```bash

```

Shows approval rate, confidence calibration buckets, and top corrections for
LLM-proposed failure classifications.

### Drift Detection

```bash
curl http://localhost:8090/admin/llm/drift
```

Compares last 7 days vs prior 7 days on latency, error rate, token usage, and
approval rate. Returns alerts when:
- Average latency increased >50%
- Error rate >5%
- Error rate doubled vs prior week
- Approval rate dropped >10 points

### Submit Feedback

```bash

  -H "Content-Type: application/json" \

```


enable tracking).

## Troubleshooting

### Pod Crashes

**Symptom**: StarGate pods in CrashLoopBackOff or Error state.

```bash
# Check pod status
oc get pods -n stargate

# Check logs
oc logs -n stargate deployment/stargate-api --tail=100

# Check events
oc get events -n stargate --sort-by='.lastTimestamp'
```

**Common causes**:

- **Database connection refused**: PostgreSQL pod not ready. Check
  `oc get pods -n stargate | grep postgres`. Ensure PVC is bound.
- **Import error**: Missing dependency. Check Containerfile includes all
  required packages.
- **Kubeconfig missing**: Scanner pods fail if `stargate-kubeconfigs` secret
  is not mounted. Verify: `oc get secret stargate-kubeconfigs -n stargate`.
- **OOM killed**: Increase memory limits in Helm values.

### Scanner Stalls

**Symptom**: Dashboard shows stale data. Health endpoint reports scanners as
"stale" (data older than 10 minutes).

```bash
# Check health
curl http://localhost:8090/health

# Check scheduler status
curl http://localhost:8090/admin/scheduler/status

# Check scan-history files
ls -lt scan-history/scan-*.json | head -5
```

**Common causes**:

- **Expired tokens**: Service account tokens expire. Regenerate kubeconfigs
  and update the `stargate-kubeconfigs` secret.
  ```bash
  oc whoami --kubeconfig=secrets/kubeconfig-ocpv06
  ```
- **Network issues**: Cluster unreachable. Check `oc cluster-info` with the
  relevant kubeconfig.
- **Worker thread crashed**: The scheduler uses exponential backoff
  (30s, 60s, 120s, max 300s) on errors. Check logs for repeated errors on
  a specific cluster.
- **Stagger drift**: All workers start with staggered offsets. If the API pod
  restarts, workers need to be restarted too. Use the
  `POST /admin/scheduler/start` endpoint.

### LLM Failures

**Symptom**: LLM-dependent features return empty responses. Classification and
summary endpoints return generic fallbacks.

```bash
# Check LLM metrics
curl http://localhost:8090/admin/llm/metrics

# Check circuit breaker state (look for error_rate)
curl http://localhost:8090/admin/llm/drift

# Check recent calls for errors
curl http://localhost:8090/admin/llm/recent?limit=10
```

**Common causes**:

- **Missing API key**: `STARGATE_LITELLM_API_KEY` not set. Check the
  `stargate-secrets` OpenShift secret.
- **LiteLLM proxy down**: Check connectivity to
  `https://litellm.example.com`.
- **Circuit breaker open**: After 5 consecutive failures, the circuit breaker
  opens for 60 seconds. The LLM is temporarily unavailable. Check error types
  in `/admin/llm/recent`.
- **Rate limiting**: LiteLLM may rate-limit calls. Check for 429 errors in
  the LLM metrics.
- **SSL certificate**: If `STARGATE_SSL_VERIFY=true` (not default), certificate
  validation may fail for self-signed certs. Set to `false` for internal
  services.

### Database Issues

**Symptom**: API returns 500 errors. Health endpoint shows database as "error".

```bash
# Check PostgreSQL pod
oc get pods -n stargate | grep postgres
oc logs -n stargate deployment/stargate-postgres --tail=50

# Check connection from API pod
oc exec -n stargate deployment/stargate-api -- python3 -c \
  "from db.database import get_engine; print(get_engine().connect().execute(__import__('sqlalchemy').text('SELECT 1')).scalar())"

# Check PVC space
oc get pvc -n stargate
```

**Common causes**:

- **PVC full**: The 20Gi PVC is full. Backup, prune old data, or expand PVC.
- **Connection pool exhausted**: Pool size is 20 with overflow 10. If >30
  concurrent connections are needed, increase `pool_size` in `db/database.py`.
- **Migration failure**: `alembic upgrade head` failed on startup. Check API
  pod logs for migration errors. Fall back to `Base.metadata.create_all()`.
- **Corrupted data**: Restore from backup:
  ```bash
  ./scripts/db-restore.sh backups/stargate-YYYYMMDD-HHMMSS.sql.gz
  ```

### Materialized View Refresh Failures

**Symptom**: Dashboard shows stale aggregate data (cluster summaries, pipeline
stages, lab evaluations) even though raw data is current.

The MV refresh thread runs every 60 seconds. Check API logs for
`"MV refresh failed"` warnings. Common cause is a database lock timeout.
Restart the API pod if persistent.

### Scan History Disk Usage

Scan history files accumulate in `scan-history/`. Each file is a few KB but
thousands can accumulate over days.

```bash
# Count files
ls scan-history/scan-*.json | wc -l
ls scan-history/babylon-*.json | wc -l

# Total size
du -sh scan-history/

# Prune files older than 3 days
find scan-history/ -name "*.json" -mtime +3 -delete
```

## Monitoring Endpoints

### Health Check

```
GET /health
```

Returns component status for API, database, and scanners. Scanners are "stale"
if data is older than 10 minutes.

### Prometheus Metrics

```
GET /metrics
```

Exposed metrics:
- `http_requests_total{method, path, status}` -- Request counter
- `http_request_duration_seconds{method, path}` -- Request latency histogram
- `llm_calls_total{endpoint, success}` -- LLM call counter
- `llm_call_duration_seconds{endpoint}` -- LLM call latency
- `llm_tokens_total{endpoint, type}` -- Token usage
- `scanner_clusters_healthy` -- Number of healthy scanner clusters

### Scan History Timeline

```
GET /admin/scan-history?limit=50
```

Returns timestamped cluster health snapshots for trend analysis.
